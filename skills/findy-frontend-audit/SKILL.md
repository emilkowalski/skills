# FINDY Frontend Audit

코드를 고치는 스킬이 아니라 현재 품질을 증거로 판정하는 감사 스킬이다. 사용자가 수정을 명시하면 감사 결과를 먼저 확정한 뒤 `findy-flet-ui` 절차로 최소 변경을 적용한다.

## 감사 절차

1. 저장소 규칙과 `docs/FRONTEND_QA.md`를 읽는다.
2. `git status --short`를 확인하고 기존 사용자 변경을 감사 대상과 분리한다.
3. `main.py`에서 실제 실행 경로를 추적하고 죽은 코드나 보관 경로를 근거로 사용하지 않는다.
4. [감사 체크리스트](references/audit-checklist.md)에 따라 코드와 렌더링을 함께 점검한다.
5. 자동 테스트 결과와 실기기 증빙을 구분한다. 실행하지 않은 항목은 `미확인`으로 둔다.
6. 문제를 사용자 영향과 출시 위험으로 정렬해 한국어로 보고한다.

## 판정 기준

- `차단`: 핵심 흐름 진입 불가, 데이터 유실, CTA 가림, 반복 크래시, 인증 우회, 복구 불가 오류.
- `높음`: 좁은 화면 잘림, 키보드 충돌, 마지막 콘텐츠 가림, 주요 상태 동기화 실패, 접근 불가능한 핵심 제어.
- `중간`: 정보 위계, 작은 글자, 불명확한 빈 상태, 과도한 밀도, 일관성 회귀.
- `낮음`: 출시를 막지 않는 광학 정렬, 미세 간격, 문구 다듬기.

`MVP 출시 가능`과 `상위 소비자 앱 수준으로 보완 필요`을 별도로 판정한다. 토스·배달의민족·당근마켓을 색이나 레이아웃 복제 대상으로 쓰지 말고, 예측 가능성·오류 복구·속도·신뢰감의 비교 기준으로만 사용한다.

## 필수 출력

1. 심각도순 발견 사항. 각 항목에 사용자 영향, 재현 조건, `file:line`, 권장 수정 범위를 넣는다.
2. 현재 수준 요약: MVP 가능 여부와 상위 소비자 앱 수준과의 거리.
3. 이미 잘된 부분. 근거 없는 칭찬은 제외한다.
4. 실행한 검증과 실행하지 못한 검증.
5. 남은 출시 차단 요소와 다음 우선순위.

문제가 없으면 명확히 그렇게 말하고 실기기·외부 연동 등 남은 검증 공백을 적는다. 같은 문제를 여러 표현으로 반복하지 않는다.

## 경계

- 분석 요청에서 코드를 수정하지 않는다.
- 실제 콘텐츠가 없는 화면을 출시 가능으로 판정하지 않는다.
- 서버가 연결되었다는 가정이 있어도 프론트의 로딩·빈 상태·오류·재시도 UX를 생략하지 않는다.
- 스크린샷은 사용자가 요청할 때만 최종 답변에 첨부한다.
- 법률, 스토어 승인, 외부 공급자 설정, 실기기 결과를 추측해 완료 처리하지 않는다.
